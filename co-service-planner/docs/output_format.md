# docs/output_format.md
# 서비스 기획 산출물 형식 & 검증 기준

> 모든 기획 산출물은 이 형식을 **반드시 준수**해야 합니다.
> 형식 위반 시 출력을 중단하고 재작성합니다.

---

## 1. 산출물 구조 (출력 순서 고정)

기획 보고서는 **아래 순서를 절대 변경할 수 없습니다**:

```
[프로젝트명] 서비스 기획 보고서
─────────────────────────────────
섹션 1. 서비스 컨셉 정의 (JTBD + 가치 제안)
섹션 2. 타겟 사용자 페르소나
섹션 3. User Journey Map
섹션 4. User Flow & Information Architecture
섹션 5. 기능 요구사항 명세 (MoSCoW + RICE)
섹션 6. MVP 정의 & Phase별 로드맵
```

---

## 2. 각 섹션별 필수 포함 항목

### 섹션 1. 서비스 컨셉 정의

필수 항목:
- **핵심 한 줄 정의** (1문장, "누가/어떤 상황에서/무엇을 위해 사용하는 서비스")
- **JTBD 문장**: "사용자가 [상황]에서 [목표]를 달성하기 위해 이 서비스를 사용한다."
- **핵심 Pain Point**: 3개 이상, 각 Pain Point는 구체적 수치 또는 사용자 행동 증거 포함
- **가치 제안 (Value Proposition)**: 경쟁 대비 차별점 명시
- **USP (Unique Selling Point)**: 3문장 이내

### 섹션 2. 타겟 사용자 페르소나

필수 항목:
- 페르소나 2~3개 (각각 가상 이름, 연령, 직업, 라이프스타일, 현재 대안, 핵심 불편함, 기대치)
- 각 페르소나에 JTBD 1문장 연결
- Market Analyst 보고서의 페르소나와 연계 명시

형식:
```
| 항목 | 페르소나 A | 페르소나 B |
|------|-----------|-----------|
| 이름 | | |
| 연령/성별 | | |
| 직업 | | |
| 라이프스타일 | | |
| 현재 대안 | | |
| 핵심 불편함 | | |
| 기대치 | | |
| JTBD | | |
```

### 섹션 3. User Journey Map

필수 항목:
- **7단계 필수**: Awareness → Consideration → Purchase → Onboarding → Core Usage → Retention → Advocacy
- 각 단계별 사용자 행동(Action), Emotional State, Pain Point, Opportunity 명시
- Emotional State는 "긍정/중립/부정" 이외에 구체적 감정 단어 사용

형식:
```
| 단계 | Awareness | Consideration | Purchase | Onboarding | Core Usage | Retention | Advocacy |
|------|-----------|--------------|---------|------------|-----------|----------|---------|
| 행동 | | | | | | | |
| Emotional State | | | | | | | |
| Pain Point | | | | | | | |
| Opportunity | | | | | | | |
```

### 섹션 4. User Flow & Information Architecture

필수 항목:
- **핵심 User Flow**: 회원가입, 핵심 기능 사용, 결제(해당 시) 3가지 이상 Flow
- **Conditional Flow**: 분기 조건 명시 (if/else)
- **Error Flow**: 실패 케이스 처리 경로
- **Information Architecture (IA)**: 전체 메뉴 트리 구조

형식:
```
[시작점] → [화면 A] → [화면 B] → (조건: Y/N) → [화면 C/D] → [종료점]

홈
├── 메뉴 1
│   ├── 서브 메뉴 1-1
│   └── 서브 메뉴 1-2
├── 메뉴 2
└── 메뉴 3
```

### 섹션 5. 기능 요구사항 명세 (MoSCoW + RICE)

필수 항목:
- **MoSCoW 분류표**: Must / Should / Could / Won't 4개 분류
- **RICE Score 계산**: Reach × Impact × Confidence / Effort
  - Reach: 월간 영향 사용자 수 (숫자)
  - Impact: 0.25 / 0.5 / 1 / 2 / 3 중 선택
  - Confidence: % (숫자)
  - Effort: 개발 인월 (숫자)
- Must-have 기능: 5개 이상

형식:
```
| 기능 | 분류 | Reach | Impact | Confidence | Effort | RICE Score | 근거 |
|------|------|-------|--------|-----------|--------|-----------|------|
| | Must | | | | | | |
```

### 섹션 6. MVP 정의 & Phase별 로드맵

필수 항목:
- **MVP 핵심 기능**: 3~5개, 각 기능의 포함 이유 명시
- **MVP 검증 지표**: 정량 목표 (예: DAU 1,000명, 리텐션 30% 이상)
- **성공 기준**: 수치 기반 Go/No-Go 판단 기준
- **Phase 로드맵**: Phase 1~3, 각 Phase마다 기간·주요 기능·목표 명시

형식:
```
| Phase | 기간 | 주요 기능 | 목표 |
|-------|------|---------|------|
| Phase 1 (MVP) | X개월 | | |
| Phase 2 (Iteration) | X개월 | | |
| Phase 3 (Scale) | X개월 | | |
```

## 3. 금지 표현 목록

아래 표현이 산출물에 포함될 경우 **즉시 재작성**:

| 금지 표현 | 대체 방향 |
|---------|---------|
| "좋은 기획이에요" / "훌륭한 아이디어" | 구체적 데이터로 뒷받침 |
| "잘 될 것 같아요" | TAM/SAM/SOM + KPI 수치로 대체 |
| "혁신적인 서비스" | 경쟁사 대비 구체적 차별점으로 대체 |
| "가능성이 높습니다" | 근거 데이터 + 리스크 병기 |
| "다양한 기능을 추가하면 좋을 것 같습니다" | RICE Score 기반 우선순위로만 제안 |
| "사용자들이 좋아할 것입니다" | 페르소나 Pain Point 연결로 대체 |
| "트렌드에 맞는 서비스" | 구체적 시장 데이터 인용 |
| "추후 논의가 필요합니다" | 현 시점 판단과 전제 조건 명시 |

## 4. 사전 검증 체크리스트 (출력 전 반드시 확인)

산출물 출력 전 아래 12개 항목을 모두 확인해야 합니다:

- [ ] **섹션 1~6 모두 포함** (하나라도 누락 시 출력 불가)
- [ ] **섹션 순서 1→6 준수** (순서 변경 불가)
- [ ] **모든 수치에 근거 명시** (Market Analyst 보고서 또는 공개 데이터 출처)
- [ ] **Pain Point 3개 이상** (각 Pain Point에 구체적 행동/수치 증거)
- [ ] **페르소나 2개 이상** (각 JTBD 연결 포함)
- [ ] **User Journey 7단계** (누락 단계 없이 모두 작성)
- [ ] **User Flow 3개 이상** (핵심 기능 + Conditional + Error Flow)
- [ ] **MoSCoW 4개 분류 모두 작성** + **RICE Score 계산 완료**
- [ ] **Must-have 기능 5개 이상**
- [ ] **MVP 검증 지표 수치 명시** (정량 목표)
- [ ] **Phase 1~3 로드맵 모두 작성** (기간 + 목표 포함)
- [ ] **금지 표현 미포함** (목록 대조 완료)
- [ ] **Market Analyst 연동 기록 작성** (decisions.md 섹션 7 참고)

---

## 5. Market Analyst 연동 필수 확인

기획 시작 전 Market Analyst 보고서에서 아래 항목이 모두 제공되었는지 확인:

- [ ] 타겟 사용자 페르소나 (최소 2개)
- [ ] 시장 Pain Point (최소 4개)
- [ ] TAM / SAM / SOM (수치 포함)
- [ ] 경쟁사 분석 (최소 3개 경쟁사)
- [ ] USP 제안
- [ ] 추천 비즈니스 모델 (BM)
- [ ] KPI 및 비즈니스 목표

**위 항목 중 하나라도 없으면 기획을 시작하지 않고 Market Analyst에 추가 분석을 요청한다.**
